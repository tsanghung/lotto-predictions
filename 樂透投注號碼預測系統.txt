## **1. 系統概述**

本專案旨在開發一套專屬個人的「樂透投注號碼預測系統」。系統需自動化化撈取、儲存並驗證台灣彩券「大樂透」與「今彩539」的歷史及最新開獎數據。核心功能為整合網路輿情、統計學與機率學等方法論，發揮 AI 模型最大效能計算出預測號碼，並於開獎日自動推播。系統將採用單頁應用程式（SPA）架構，並完全佈署於個人 GitHub 環境。

## **2. 系統架構規劃（由 AI 評估與設計）**

鑑於本系統僅供單人使用、無大流量與大容量儲存需求，且需佈署於 GitHub，請 Claude Code 採用以下架構方向進行開發：

1. **前端架構**：採用 HTML / CSS / JavaScript 原生開發，或輕量級框架（如 Vue.js 或 React）。必須為響應式網頁設計（RWD），確保在電腦與手機等不同解析度與平台上皆能正常顯示。
2. **後端與自動化任務**：因 GitHub Pages 不支援動態後端伺服器，需採用 GitHub Actions 作為伺服器less的執行環境，負責執行 Python 或 Node.js 腳本進行數據爬蟲、演算法運算與推播觸發。
3. **資料庫與儲存**：採用本地端輕量級資料庫（如 SQLite）或將整理好的歷史數據儲存為 JSON / CSV 格式，存放於 GitHub 儲存庫中，由 GitHub Actions 每日更新並 Commit。
4. **佈署環境**：前端頁面透過 GitHub Pages 託管，後端邏輯與排程透過 GitHub Actions 執行。

## **3. 核心功能需求**

### **A. 歷史數據庫建立與驗證（核心基礎）**

1. **數據撈取**：撈取歷史至今所有「大樂透」與「今彩539」的開獎紀錄。
2. **數據來源標示**：必須明確標示數據來源為「台灣彩券官方網站」或可信之第三方歷史開獎數據庫。嚴禁任何編造或產生幻覺的假數據。
3. **雙重驗證機制**：本地端資料庫初步建立完成後，系統必須具備自動化重複比對邏輯，重新與來源端核對一次所有歷史數據的正確性，確保資料庫 100% 準確。

### **B. 每日資料庫更新**

1. **排程更新**：每天根據台灣彩券公司公布的時程，自動撈取最新一期的開獎號碼。
2. **本地紀錄**：將最新號碼寫入本地端資料庫或數據檔案中，維持數據庫的即時性。

### **C. 預測演算法與 AI 效能極限榨取**

1. **資訊檢索**：每次推論前，系統需具備搜尋網路（或利用內建搜尋工具）的能力，檢索關於樂透號碼分析、預測的文章、評論與報導。
2. **多維度分析**：結合歷史開獎號碼的大數據，導入統計學（如：冷熱門號碼、遺漏值分析）、機率學、自相關分析 (Autocorrelation)、回歸模型預測、蒙地卡羅模擬 (Monte Carlo Simulation)等方法論。
3. **AI 推理極限**：綜合上述即時網路資訊與歷史統計特徵，發揮 LLM 模型的最大效能與深度思考推理能力，產出當期投注號碼的推論過程與最終預測結果。

### **D. 預測號碼推播機制**

1. **觸發時間**：依據「大樂透」（每週二、五）與「今彩539」（每週一至週六）的開獎日期，於開獎日當天中午 12 點準時觸發。
2. **推播方式**：整合通訊軟體 API（如 LINE Notify 或 Telegram Bot），將 AI 深度分析後產出的投注預測號碼傳送至指定帳號。

請 Claude Code 在執行本規格書進行開發時，嚴格遵守以下防錯機制：

1. **狀態回報格式**：開發過程中若遇到任何報錯、相依性衝突或預期外的邏輯阻礙，強制中斷執行，並以以下格式回報：
    - `Status`: [Error / Warning]
    - `Root Cause`: [詳細分析錯誤發生的根本原因]
    - `Suggested Fix`: [提出具體可行的修正方案或程式碼]
2. **不可逆動作確認**：在執行任何可能覆蓋大量檔案、刪除資料庫或上傳（Push）至 GitHub 等不可逆的破壞性動作前，必須先輸出當前假設並停頓，等待確認後方可繼續執行。
